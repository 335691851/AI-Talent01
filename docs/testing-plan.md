# AI Talent 本地可用性测试方案 v0.3（待确认）

## 1. 测试目标

本次测试目标聚焦“本地项目可用性和易用性”，不以生产级压测、复杂 CI/CD 或多环境兼容为主。测试要证明 AI Talent 在本机开发环境中各个关键环节可以顺畅跑通，并且页面对用户有清晰提示。

- 核心业务流程本地可用：管理员导入员工、员工维护 AI 档案、员工完成 10 轮 AI 自评估、管理员查看结果并检索人才。
- 页面易用性达标：每个关键操作都有明确的加载、进度、成功、失败、空状态提示。
- 数据安全可控：管理员不能查看完整 Chat 对话；员工只能操作自己的档案和评估；Supabase RLS 或等效访问控制保护关键数据。
- AI 与向量检索可靠：DeepSeek 流式评估可用，Cloudflare embedding 可写入 Supabase pgvector，管理员语义检索可返回候选员工。
- 本地环境易启动：`.env.local`、数据库迁移、向量化 backfill、dev server、测试命令路径清晰。

本轮不优先覆盖：

- 多租户、多企业、部门级权限。
- 大规模并发、压测、成本监控。
- 复杂生产发布流水线。
- 多浏览器矩阵测试。

## 2. 测试分层

| 层级 | 工具 | 覆盖范围 | 运行时机 |
| --- | --- | --- | --- |
| 单元测试 | Vitest | 纯函数、解析逻辑、Prompt、结构化检索、向量格式化 | 每次开发后 |
| 服务端逻辑测试 | Vitest | Server Action / Route Handler 的关键分支，必要时 mock 外部服务 | 核心接口变更后 |
| 端到端测试 | Playwright | 本地登录、导入、档案编辑、AI 自评估、管理员检索、权限跳转 | 需求确认后实现，本地验收前必跑 |
| 数据库安全验证 | SQL 脚本 / Vitest 辅助 | RLS、权限、pgvector 表、检索函数、embedding 数据量 | migration 后 |
| 手工验收 | 浏览器 | 视觉体验、提示文案、复杂异常场景 | 发布前 |

## 4. Playwright 本地端到端测试方案

| 角色 | 账号 | 密码 | 说明 |
| --- | --- | --- | --- |
| 管理员 | `admin` | `admin` | 本地初始主管理员 |
| 员工 | Excel 导入手机号 | `88888888` | 用导入数据创建 |

建议测试数据：

- 使用专用测试员工，工号建议为 `TEST-1001`。
- 每次测试前清理或覆盖同一工号，避免污染真实员工数据。
- 本地可用性测试优先使用演示 AI 响应，确保 10 轮流程稳定可复现。
- 真实 DeepSeek 流式调用作为单独 smoke test，验证配置和链路可用即可。

- `npm.cmd run test` 通过。

- `npm.cmd run lint` 通过。

- 至少 1 名测试员工完成：Excel 导入、员工登录、档案编辑、10 轮 AI 自评估、管理员检索。
- 向量化 backfill 成功，`employee_embeddings` 有数据。
- 管理员不能查看完整 Chat 对话。
- 所有核心页面都有加载、进度、成功、失败或空状态提示。
- 本地浏览器无白屏、无明显控制台运行时错误。
